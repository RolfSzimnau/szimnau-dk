Jemand in einer dänischen Facebook-Gruppe erwähnte beiläufig, dass man Home Assistant so einrichten könne, dass man beim Betreten eines Geschäfts benachrichtigt wird, wenn dort Artikel auf der eigenen Einkaufsliste angeboten werden. Interessant. Ich scrollte weiter.

Eine Woche später stand ich selbst im Supermarkt — eTilbudsavis in einem Tab, meine Notizen in einem anderen — und wusste nicht mehr, welche Angebote noch gültig waren. Der Facebook-Kommentar fiel mir wieder ein.

## Die Integration — CagosDk auf GitHub

Der Ausgangspunkt ist eine HACS-Integration von [CagosDk](https://github.com/CagosDk/hacs-etilbudsavis). Sie wird über HACS → Integrationen → Benutzerdefinierte Repositories hinzugefügt. Die Integration erzeugt eine `todo`-Entität für jede eTilbudsavis-Einkaufsliste — bei mir `todo.min_indkobsliste`. Sie erscheint dann als natives HA-Todo-Karte und in der Companion-App.

Das Zwei-App-Problem war damit gelöst. Die Standardansicht zeigt aber nur Artikelnamen — kein Geschäft, kein Preis, kein Datum. Bei fünfzehn Artikeln aus vier verschiedenen Geschäften hilft das wenig, wenn es schnell gehen muss.

## Nach Geschäft sortiert, Preis in der Beschreibung

eTilbudsavis liefert pro Artikel Geschäftsinformationen und Angebotsdaten über die API. Die `todo.py` der Integration kann diese Felder auf das `description`-Feld des Todo-Items abbilden. Mit einer kleinen Anpassung zeigt die Beschreibung jetzt:

```
Netto – 12,95 kr. · til 12/6
```

Die Liste ist sortiert: alle Netto-Artikel zusammen, dann ABC Lavpris, dann Bilka. Abgehakte Artikel landen unabhängig vom Geschäft am Ende.

Der Sortierschlüssel in der `todo_items`-Property lautet `(completed, store.lower(), summary.lower())`. Eine Zeile. Preis und Datum werden nur angehängt, wenn die API sie zurückgibt — kein Absturz, kein leeres Feld wenn sie fehlen.

## Zonen und Proximity-Benachrichtigungen

Das ist der Teil aus dem Facebook-Kommentar. Home Assistant hat einen `zone`-Entitätstyp — man legt einen Radius um eine Adresse fest und die Companion-App trackt, wann das Telefon diese Zone betritt oder verlässt. Ich habe Zonen für alle relevanten Geschäfte angelegt: Min Kobmand in Jordrup, ABC Lavpris in Egtved und Bilka, Lidl, Føtex sowie zwei Netto-Filialen in Kolding.

Eine Automation löst aus, wenn mein Pixel 10 oder das Pixel 9a meiner Frau eine dieser Zonen betritt:

```yaml
trigger:
  - platform: zone
    entity_id:
      - device_tracker.pixel_10
      - device_tracker.pixel_9a
    zone: zone.bilka
    event: enter
```

Die Automation ordnet jede Zone einem Geschäftsnamen in einem `variables`-Block zu und ruft dann `todo.get_items` ab. Artikel, deren `description` mit dem Geschäftsnamen beginnt, werden herausgefiltert:

```yaml
variables:
  butik_lookup:
    zone.bilka: "Bilka"
    zone.netto_hylkedalvej: "Netto"
    zone.netto_stadionvej: "Netto"
  butik_navn: "{{ butik_lookup[trigger.zone] | default('') }}"
```

```yaml
- variables:
    varer: >
      {% set alle = todo_svar['todo.min_indkobsliste']['items'] | default([]) %}
      {% set ns = namespace(liste=[]) %}
      {% for vare in alle %}
        {% if vare.description | default('') | regex_match(butik_navn) %}
          {% set ns.liste = ns.liste + [vare.summary] %}
        {% endif %}
      {% endfor %}
      {{ ns.liste }}
- condition: template
  value_template: "{{ varer | count > 0 }}"
- data:
    title: "🛒 {{ butik_navn }} – {{ varer | count }} Artikel auf der Liste"
    message: "{{ varer | join('\n') }}"
  action: notify.alle_enheder
```

`regex_match` matcht vom Anfang des Strings, sodass „Netto" auf „Netto – 12,95 kr. · til 12/6" passt, ohne einen contains-Check zu brauchen. Beide Netto-Filialen in Kolding lösen dieselbe Benachrichtigung aus.

Das erste Mal funktionierte es vor dem Netto auf dem Hylkedalvej. Telefon vibriert. Zwei Artikel auf der Liste. Ich wusste es eigentlich — ich hatte die Liste selbst geschrieben. Aber es auf dem Display zu sehen genau in dem Moment, in dem man das Geschäft betritt, ist etwas anderes als daran zu denken, die App zu öffnen.
